# Blaze Phase 3c – Extended Library: Dynamic Dispatch Helpers (`std::dynamic`)

> **Goal:** Implement the `std::dynamic` module exactly as specified.  This module provides utilities for working with `dyn Trait` objects, including downcasting, type checking, and constructing trait objects from concrete types.  It serves as the runtime support layer for dynamic dispatch beyond what is built into the compiler.

---

## 1. `DynTrait` Type

### 1.1 Definition

```
pub struct DynTrait<T: ?Sized> {
    data: *mut u8,
    vtable: *const VTable,
}
```

- `DynTrait` is the internal representation of a `dyn Trait` value.  It is not constructible directly by users; instead, the compiler automatically coerces concrete references or owned values into `dyn Trait` by constructing a `DynTrait` pair.
- The `data` pointer points to the concrete value (or the value itself if the trait is `?Sized` – for a reference, `data` is the pointer to the concrete type; for an owned object, `data` is a pointer to the heap‑allocated object).
- The `vtable` pointer points to a vtable containing function pointers for all methods of the trait.

### 1.2 Raw Creation (unsafe)

```
impl DynTrait<T: ?Sized> {
    pub unsafe fn from_raw_parts(data: *mut u8, vtable: *const VTable) -> Self;
    pub fn data_ptr(&self) -> *mut u8;
    pub fn vtable_ptr(&self) -> *const VTable;
}
```

- `from_raw_parts` allows library code to construct a `DynTrait` manually (useful for FFI or custom allocators).  The caller must ensure the pointer and vtable are valid.
- `data_ptr` and `vtable_ptr` return the components.

---

## 2. VTable

### 2.1 `VTable` Struct

```
pub struct VTable {
    drop: unsafe fn(*mut u8),          // destructor
    size: usize,
    align: usize,
    methods: *const MethodTable,
}
```

- The vtable contains the destructor, the size and alignment of the concrete type, and a pointer to an array of method pointers (`MethodTable`).
- `MethodTable` is an opaque struct defined internally by the compiler; it is not accessed directly by users.  The compiler populates it with function pointers in the order of the trait’s methods.

### 2.2 Method Table (Internal)

```
#[repr(C)]
pub struct MethodTable {
    // variable‑length array of function pointers
    // The compiler knows the layout; library code does not index into it.
}
```

- The `MethodTable` is never constructed by library code; the compiler generates the appropriate vtable for each concrete type implementing a trait.

---

## 3. Downcasting

The module provides functions to attempt conversion of a `dyn Trait` object back to a concrete type, using the trait’s `Any`‑like supertrait support.  For now, we support downcasting only for `dyn Any`.

### 3.1 `Any` Trait

```
pub trait Any: 'static {}
```

- `Any` is automatically implemented for all non‑generic types that satisfy `'static`.  It is a marker trait.
- The compiler provides a blanket implementation.

### 3.2 `downcast_ref` and `downcast_mut`

```
pub fn downcast_ref<T: Any>(obj: &dyn Any) -> Option<&T>;
pub fn downcast_mut<T: Any>(obj: &mut dyn Any) -> Option<&mut T>;
```

- Returns a reference to the concrete value if the trait object is indeed of type `T`.  Uses a type‑ID check against the vtable’s stored type ID.  The type ID is a unique per‑type hash generated by the compiler.

### 3.3 Type ID

```
pub struct TypeId(u64);

impl TypeId {
    pub fn of<T: Any>() -> TypeId;
}
```

- The `TypeId` is an opaque value that uniquely identifies the type.  The compiler generates a call to `TypeId::of::<ConcreteType>()` when building the vtable, and the stored ID is compared at runtime in `downcast_ref`.

---

## 4. `impl Trait` Support

`impl Trait` in argument position is compiled as a separate monomorphised function (as per Phase 2b).  The `std::dynamic` module does not provide additional runtime support; it only supports `dyn Trait` objects.

---

## 5. Implementation Notes

- The `VTable` struct and method table are generated by the compiler during monomorphisation of trait implementations.
- `downcast_ref` and `downcast_mut` rely on the vtable containing a pointer to a `TypeId` at a known offset.  The compiler inserts this when building vtables for types implementing `Any`.
- The `TypeId` is a hash of the type’s mangled name, computed at compile time; collisions are possible but astronomically unlikely for valid programs.  We will use a 64‑bit FNV‑1a hash of the fully qualified type name.

---

## 6. Testing

- **Creating a `dyn Trait`:** Take a concrete type implementing a trait, coerce to `dyn Trait`, verify that calling a method via the trait object works correctly (by comparing output).
- **Downcasting:** Create a `dyn Any` from a concrete type, downcast back, verify the reference is valid and points to the same value.
- **Incorrect downcasting:** Attempt to downcast to a different type, verify it returns `None`.
- **TypeId uniqueness:** Compare `TypeId::of::<i32>()` and `TypeId::of::<String>()`, ensure they differ.
- All tests must pass before moving to the next module.
