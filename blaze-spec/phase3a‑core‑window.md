# Blaze Phase 3a – Core Library: Sliding Windows (`std::window`)

> **Goal:** Implement the `std::window` module exactly as specified.  This module provides zero‑cost abstractions for iterating over fixed‑size overlapping or strided windows of slices, with deep integration into Blaze’s structure‑of‑arrays (SoA) layout system.

---

## 1. Core Traits

### 1.1 `Windowable`

```
pub trait Windowable<const N: usize> {
    type WindowIter: Iterator<Item = &[Self::Item; N]>;
    fn window(&self) -> Self::WindowIter;
}
```

- Implemented by contiguous collections (e.g., slices, `Vec<T>`, arrays) to provide a sliding window iterator of `N` elements.
- The iterator yields overlapping slices of length `N`, starting at index 0 and advancing by 1.

### 1.2 `SoAWindowable`

```
pub trait SoAWindowable<const N: usize> {
    type WindowIter: Iterator<Item = WindowedView<Self, N>>;
    fn window_soa(&self) -> Self::WindowIter;
}
```

- Automatically implemented by the compiler for any struct annotated with `@layout(soa)` when its collection is instantiated (e.g., `Vec<S>` or `&[S]`).
- Provides per‑field, cache‑friendly window access.

---

## 2. Iterator Types

### 2.1 `Window<T, const N: usize>`

```
pub struct Window<T, const N: usize> {
    ptr: *const T,
    len: usize,
}
```

- Holds a pointer to the slice and its length.
- Implements `Iterator<Item = &[T; N]>`.

Implementation:

```
impl<T, const N: usize> Window<T, N> {
    pub fn new(slice: &[T]) -> Self {
        Window { ptr: slice.as_ptr(), len: slice.len() }
    }
}
impl<T, const N: usize> Iterator for Window<T, N> {
    type Item = &[T; N];
    fn next(&mut self) -> Option<&[T; N]> {
        if self.len < N { return None; }
        unsafe {
            let win = &*(self.ptr as *const [T; N]);
            self.ptr = self.ptr.add(1);
            self.len -= 1;
            Some(win)
        }
    }
}
```

- No bounds checks inside the iterator; the check `len < N` ensures safety.

### 2.2 `WindowMut<T, const N: usize>`

```
pub struct WindowMut<T, const N: usize> {
    ptr: *mut T,
    len: usize,
}
```

- Yields mutable slices `&mut [T; N]`.
- Linear: only one active mutable window at a time (the iterator borrows the slice exclusively; the compiler’s affine logic ensures the parent slice cannot be used while the iterator exists, but since the iterator itself is a value, the uniqueness is not enforced by the type system except by convention.  For safety, `WindowMut::new` takes `&mut [T]` and returns an iterator that borrows it, so the original slice cannot be used while the iterator is alive, enforced by NLL.  We'll require that the passed slice is borrowed mutably for the iterator lifetime.)
- Implementation:

```
impl<T, const N: usize> WindowMut<T, N> {
    pub fn new(slice: &mut [T]) -> Self {
        WindowMut { ptr: slice.as_mut_ptr(), len: slice.len() }
    }
}
impl<T, const N: usize> Iterator for WindowMut<T, N> {
    type Item = &mut [T; N];
    fn next(&mut self) -> Option<&mut [T; N]> {
        if self.len < N { return None; }
        unsafe {
            let win = &mut *(self.ptr as *mut [T; N]);
            self.ptr = self.ptr.add(1);
            self.len -= 1;
            Some(win)
        }
    }
}
```

### 2.3 `StridedWindow<T, const N: usize, const S: usize>`

- Provides windows of size `N` with a stride of `S` elements between successive windows.
- Holds a pointer and length; advances by `S` on each step.

```
pub struct StridedWindow<T, const N: usize, const S: usize> {
    ptr: *const T,
    len: usize,
}

impl<T, const N: usize, const S: usize> StridedWindow<T, N, S> {
    pub fn new(slice: &[T]) -> Self {
        StridedWindow { ptr: slice.as_ptr(), len: slice.len() }
    }
}
impl<T, const N: usize, const S: usize> Iterator for StridedWindow<T, N, S> {
    type Item = &[T; N];
    fn next(&mut self) -> Option<&[T; N]> {
        if self.len < N { return None; }
        unsafe {
            let win = &*(self.ptr as *const [T; N]);
            self.ptr = self.ptr.add(S);
            self.len = self.len.saturating_sub(S);
            Some(win)
        }
    }
}
```

Note: If `S` is smaller than `N`, windows will overlap; if `S` is larger, gaps appear.  This is intentional.

### 2.4 `WindowSoA<S, const N: usize>`

- An iterator over SoA‑transformed collections of a struct `S` (with `@layout(soa)`).
- Yields `WindowedView<S, N>` values, which are compiler‑generated types providing per‑field slice access.
- This struct is generated by the compiler, not hand‑written.  Its implementation depends on the internal representation of SoA arrays.  For our library, we will provide a placeholder type and a code‑generation attribute; the actual implementation is deferred to the compiler’s code‑gen phase.

---

## 3. `WindowedView<S, N>`

- This is a **compiler‑synthesized type** specific to each struct `S` that has `@layout(soa)`.
- For each field `field: Type` of `S`, `WindowedView<S,N>` contains a method:
  ```
  pub fn field(&self) -> &[Type; N];
  ```
  (and for mutable windows, `&mut [Type; N]`).
- Internally, the view holds a pointer to the start of each SoA field array; the method returns a slice directly into that array without any overhead.

Because this type is generated, the `std::window` module does not define it directly.  The `SoAWindowable` trait's associated types will be the concrete generated iterator that yields the appropriate `WindowedView`.  For testing, we will use a mock struct that simulates the SoA layout.

---

## 4. Integration with Window Hints (`@slide`, etc.)

The `@slide(N)` attribute on a function parameter (of slice type) tells the compiler that the function will be called repeatedly with a sliding window of size `N`.  The compiler can then optimize loops using this knowledge.  However, this attribute is not part of the library; it’s a compiler hint.  For completeness, we document:

- `@slide(N)` – enables unrolling and vectorisation tailored to window width.
- `@window_nobounds` – tells compiler to omit bounds checks on window iterations (unsafe if misused).
- `@window_accumulate` – marks an associative reduction over sliding windows; the compiler may replace with O(N) prefix‑sum.

These are implemented in the compiler, not in the library, but the window iterators are the target.

## 5. Testing

- **`Window`:** Create a slice of integers, iterate over windows of size 2 and 3, verify overlapping sequences.
- **`WindowMut`:** Mutate elements through windows and verify original slice changed.
- **`StridedWindow`:** Test with stride > 1, stride < N, and stride = 1.
- **`SoAWindowable`:** For a struct with `@layout(soa)`, verify that `window_soa` yields views that access the correct elements. (This requires a compiler feature that may not be fully implemented in this phase; we will test manually once SoA transformation is available.)

All tests must pass before proceeding to Phase 3b.
