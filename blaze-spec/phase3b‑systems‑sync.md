# Blaze Phase 3b – Systems Library: Synchronization (`std::sync`)

> **Goal:** Implement the `std::sync` module exactly as specified.  This module provides the channel types (`Sender`, `Receiver`) and the capability type (`Capability<A>`) used for actor message passing.

---

## 1. `Sender<T>`

### 1.1 Struct

```
pub struct Sender<T> {
    // reference‑counted pointer to inner queue
}
```

A `Sender<T>` is a cloneable, reference‑counted handle that can send values of type `T` into a channel.  It can be safely cloned and shared across actors and threads (satisfies `Send` and `Sync` automatically because the underlying queue is thread‑safe and the type does not own any non‑`Sync` data).

### 1.2 Methods

```
impl<T> Sender<T> {
    pub fn send(&self, value: T) -> Result<(), SendError<T>>;
}
```

- `send(value)`: attempts to enqueue `value` into the channel.  If the receiver has been dropped, the send fails and returns `Err(SendError(value))`.  Otherwise, returns `Ok(())`.  This is a non‑blocking operation (if the channel buffer is full, the sender may block briefly or return an error depending on the channel configuration; the default unbounded channel never blocks, but a bounded channel created via `channel_with_capacity` will block the sender thread when full – however, actors should not block; we will implement channels as unbounded for now and later add backpressure options).

For Phase 3b, we only provide unbounded channels.

### 1.3 Trait Implementations

```
impl<T> Clone for Sender<T> {
    fn clone(&self) -> Self;
}
```

- Increments the internal reference count and returns a new `Sender` pointing to the same channel.

```
impl<T> Dispose for Sender<T> {
    fn dispose(&mut self);
}
```

- Decrements the reference count; when the last sender is disposed, the channel’s sender side is closed, but the receiver can still drain remaining messages.

---

## 2. `Receiver<T>`

### 2.1 Struct

```
pub struct Receiver<T> {
    // owns the receiving end of the channel
}
```

`Receiver<T>` is a linear type (not cloneable) that receives values from the channel.  It can be used as an iterator.

### 2.2 Methods

```
impl<T> Receiver<T> {
    pub fn recv(&mut self) -> Option<T>;
}
```

- Attempts to dequeue a value.  Returns `Some(value)` if a message is available, or `None` if all senders have been dropped and the channel is empty.

### 2.3 Trait Implementations

```
impl<T> Iterator for Receiver<T> {
    type Item = T;
    fn next(&mut self) -> Option<T> {
        self.recv()
    }
}
```

- Allows `for` loops to drain messages.

```
impl<T> Dispose for Receiver<T> {
    fn dispose(&mut self);
}
```

- Drops the receiver; any unread messages are disposed (if `T: Dispose`).  Future sends to the channel will fail.

---

## 3. Channel Creation

```
pub fn channel<T>() -> (Sender<T>, Receiver<T>);
```

- Creates an unbounded channel pair.  The sender can be cloned, the receiver is unique.

```
pub fn channel_with_capacity<T>(capacity: usize) -> (Sender<T>, Receiver<T>);
```

- Creates a bounded channel with the given buffer size.  Sends will block when the buffer is full (in an actor context, blocking is allowed because the sender’s actor will be suspended until space is available).  This function is provided for completeness and may be used in non‑actor contexts.

For Phase 3b, `channel_with_capacity` is optional but recommended.

---

## 4. `SendError<T>`

```
pub struct SendError<T>(pub T);
```

- Returned by `Sender::send` when the receiver has been dropped.  Contains the value that could not be sent.

---

## 5. `Capability<A>`

### 5.1 Struct

```
pub struct Capability<A> {
    // opaque handle to an actor
}
```

`Capability<A>` represents the ability to send messages to an actor of type `A`.  It is created by `spawn()` (the compiler generates `spawn` as part of the actor system).  The exact content of a `Capability` is a pointer to the actor’s mailbox and a reference to its supervisor.  It is cloneable and can be sent between actors.

### 5.2 Methods

`Capability<A>` itself has no inherent methods except those generated by the compiler for each `@message` method of actor `A`.  These generated methods are placed in the same module as the actor definition, but they operate on `Capability<A>`.  For example:

```
// for actor Counter with @message fn inc(&mut self, n: i32) -> i32
// the compiler generates:
pub fn inc(cap: &Capability<Counter>, n: i32) -> Request<i32>;
```

The `std::sync` module does not need to define these; they belong to the actor’s module.  However, `Capability<A>` must be defined here.

### 5.3 Trait Implementations

```
impl<A> Clone for Capability<A> {
    fn clone(&self) -> Self;
}
```

- Capabilities can be freely duplicated (reference‑counted internally).  This allows multiple senders to message the same actor.

```
impl<A> Dispose for Capability<A> {
    fn dispose(&mut self);
}
```

- Decrements the reference count.  When the last capability is dropped, the actor is notified that no more external messages will arrive; the actor may shut down after processing remaining messages (if it has no other capabilities).

---

## 6. Implementation Notes

- Under the hood, the channel uses a lock‑free linked list of nodes (or a ring buffer) protected by atomic operations.  The send side pushes a node, the receive side pops.  Reference counting on the sender uses an atomic counter.
- `Capability` holds a pointer to the actor’s mailbox (which is the same channel used for messages) plus a reference to the actor’s supervisor.  The mailbox is essentially a `Receiver<ActorMessage>` where `ActorMessage` is an enum of all possible message types for that actor, auto‑generated by the compiler.

---

## 7. Testing

- **Channel:** Create a sender/receiver pair, send a value, verify it is received.  Clone the sender, send from both, receive all messages.  Drop the receiver, verify that send fails with `SendError`.
- **Capability:** (Since actor spawn is already implemented in Phase 2d, we can test that a `Capability` can be cloned and that sending a message via the capability works.)
- **Disposal:** Ensure that when all senders are dropped, the receiver yields `None` after draining; when the receiver is dropped, pending sends fail.
- Thread‑safety (maybe asynchronously): spawn multiple actors sending on the same channel and verify no data races (this test passes because channels are lock‑free).

All tests must pass before moving to the next module.
