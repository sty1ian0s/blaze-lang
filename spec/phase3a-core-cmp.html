Blaze Phase 3a – Core Library: Comparison

Previous: Clone | Next: Collections

Goal

Implement the std::cmp module: traits for equality, ordering, and approximate equality, along with the Ordering enum.

1. Traits

1.1 PartialEq

pub trait PartialEq {
    fn eq(&self, other: &Self) -> bool;
}

Must be symmetric and transitive. Used by ==.

1.2 Eq

pub trait Eq: PartialEq {}

Marker trait for true equivalence (reflexive, symmetric, transitive). All integer types and bool implement Eq. Floating‑point types do not (due to NaN).

1.3 Ordering

pub enum Ordering {
    Less,
    Equal,
    Greater,
}

1.4 PartialOrd

pub trait PartialOrd: PartialEq {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering>;
}

Returns None for incomparable values (like NaN). Used by <, >, etc.

1.5 Ord

pub trait Ord: Eq + PartialOrd {
    fn cmp(&self, other: &Self) -> Ordering;
}

Total order. All integers and bool implement Ord.

1.6 ApproxEq

pub trait ApproxEq {
    fn approx_eq(&self, other: &Self, epsilon: Self) -> bool;
}

Used by =~ operator. Default epsilon: 4 * EPSILON of the type. Implemented for f32, f64.

2. Standard Implementations

  • All integer types: PartialEq, Eq, PartialOrd, Ord.
  • All float types: PartialEq, PartialOrd, ApproxEq (no Eq/Ord).
  • bool, char, (): all four (where applicable).
  • &T: delegates to T::eq (value comparison).

3. Derivation

@derive(PartialEq) generates equality by comparing all fields; @derive(Eq) auto‑applied if all fields are Eq; @derive(PartialOrd) and @derive(Ord) generate lexicographic ordering.

4. Testing

  • Primitive types: equality and ordering edge cases (MIN, MAX, NaN, infinity).
  • ApproxEq: compare with various epsilons; ensure =~ works.
  • Manual implementations for custom structs.
  • Type check: f64 cannot be used where Eq required.
