Blaze Phase 4 – Ecosystem: Accessibility

Previous: L10n | Next: Arch

Goal

Provide platform‑specific backend for translating semantic accessibility tree to native APIs (AT‑SPI, UIA, NSAccessibility).

1. Core Trait

pub trait AccessibilityBackend: Send + Sync {
    fn update(&mut self, nodes: &[AccessibilityNode]);
    fn flush(&mut self);
}

2. AccessibilityNode (re‑exported from blaze‑gui) 

pub struct AccessibilityNode {
    pub id: u64,
    pub role: AccessibilityRole,
    pub label: String,
    pub value: Option<String>,
    pub description: Option<String>,
    pub actions: Vec<AccessibilityAction>,
    pub bounding_rect: Rect,
    pub parent_id: Option<u64>,
    pub children_ids: Vec<u64>,
}
pub enum AccessibilityRole { Button, Slider, CheckBox, StaticText, Image, Link, List, ListItem, Heading, Window, Dialog, Menu, MenuItem, TabList, Tab, Tree, TreeItem, Custom(String) }
pub struct AccessibilityAction { pub name: String, pub action_type: ActionType }
pub enum ActionType { Activate, Increment, Decrement, Toggle, ScrollIntoView }

3. Backend Implementations

  • AtSpiBackend (Linux) – D‑Bus AT‑SPI2.
  • UiaBackend (Windows) – UI Automation COM.
  • NsAccessibilityBackend (macOS) – NSAccessibility.
  • NullBackend – no‑op for unsupported platforms.

4. Integration

blaze‑gui::App owns a Box<dyn AccessibilityBackend>; updates after each UI rebuild.

5. Testing

  • Null backend: no‑op.
  • Mock backend: capture nodes and verify roles/labels.
  • Platform‑specific: manual with screen reader.
