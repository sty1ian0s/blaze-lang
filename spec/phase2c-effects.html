Blaze Phase 2c – Effect System & Purity

Previous: Phase 2b | Next: Phase 2d – Concurrency

Goal

Implement the effect system: tracking effect sets, inferring minimal effect sets for each function, effect polymorphism, purity detection, and auto‑parallelisation of pure loops.

1. Effect Sets

1.1 Built‑in Effects

  • alloc – dynamic memory allocation.
  • io – I/O (implies alloc).
  • parallel – may execute in parallel.
  • gpu – uses GPU (implies parallel).
  • hal – hardware abstraction (implies io).

1.2 Effect Implications

io  ⇒ alloc
gpu ⇒ parallel
hal ⇒ io

1.3 Effect Annotations

Functions may declare effect set: fn foo() -> i32 with io. If omitted, compiler infers minimal set. Declared must be superset of inferred.

2. Effect Inference Algorithm

2.1 Bottom‑Up Fixed‑Point Analysis

  • Initialize all functions to ∅.
  • Iterate: add local effects (alloc, calls, for pure body ⇒ parallel).
  • Propagate across call graph until no changes.
  • Lattice height ≤ 5, so ≤ 5 iterations.
  • Panics, todo!, unreachable! add no effect.

2.2 Purity

Function is pure if inferred effect set is empty.

3. Effect Polymorphism

fn map<T, effect E>(v: Vec<T>, f: fn(T) -> T with E) -> Vec<T> with E;

Effect variable E binds to closure’s effect set. Inference unifies.

4. Auto‑Parallelisation of Pure Loops

A for loop is automatically parallelised if:

  • Body is pure (effect set ∅).
  • No explicit seq for modifier.
  • Not inside @comptime.

Compiler emits parallel version using runtime thread pool; adds parallel effect to enclosing function.

5. Code Generation & Runtime Integration

No runtime effect checks. Parallelised loops call blaze_parallel_for. Deterministic because body is pure.

6. Testing

  • Inference tests (allocation, I/O, calls).
  • Explicit annotation superset checks.
  • Effect polymorphism instantiation.
  • Purity enforcement.
  • Auto‑parallelisation (verify parallel code).
  • seq for sequential fallback.
  • Complexity: fixed‑point termination ≤ 5 passes; test with deep call graph.
