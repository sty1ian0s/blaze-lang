Appendix C – Closure Disambiguation

Previous: Appendix B | Next: Appendix D

Status

Normative. Defines the exact lexical and syntactic disambiguation rules for closure expressions in Blaze.

C.1 Lexical Introduction

A closure expression begins with the token | (U+007C, VERTICAL LINE). This token is also used as the bitwise OR operator, but the parser distinguishes the two contexts unambiguously.

C.2 Syntax

closure_expr = "|" param_list "|" expr
  • After |, the parser expects a param_list (identical to function parameter lists, but with optional type annotations).
  • The token | separates the parameter list from the closure body.
  • The body is an arbitrary expression, which may be a block expression { … }.

C.3 Parameter List

The parameter list may be empty, written as ||: 

|| expr
|x| expr
|x, y| expr
|x: i32, y: i32| { x + y }

Types on parameters are optional. If omitted, the compiler infers them from the closure body.

C.4 Disambiguation from Bitwise OR

The closure syntax is unambiguous because:

  • A bitwise OR expression always has a left operand. At the start of an expression (or after an opening parenthesis, brace, or operator that expects an operand), | cannot be a binary operator.
  • After the closing | of a closure, the parser expects an expression; if it sees another |, that would be part of the next expression (e.g., |x| x | y – the second | is the bitwise OR operator).

Thus, the parser never needs more than one token of lookahead to decide whether a | begins a closure.

C.5 Examples

let add_one = |x| x + 1;
let sum = |a, b| a + b;
let run = || { println("Hello"); };
let complex = |x: i32, y: i32| { let z = x + y; z * z };
let bitwise = a | b;   // not a closure, because 'a' is a left operand
