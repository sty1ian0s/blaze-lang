Blaze Phase 2f – Attributes and Zero‑Cost Abstractions

Previous: Phase 2e | Next: Phase 3a – Core Builtins

Goal

Implement all remaining attributes: built‑in expansions, conditional compilation, parser‑facing ZCAs, comptime‑expanded macros, domain‑specific bundles, and annotation attributes (including negative space and hardware attributes).

1. Built‑in Expansions

1.1 @data

Expands to @copy + @derive(Debug, Binary, Default, PartialEq, PartialOrd). Fails if type cannot be @copy.

1.2 @derive(trait_list)

Generates impl blocks for listed traits. Supported: Debug, Binary, Default, PartialEq, PartialOrd, Clone, Display, From<T>, Into<T>, AsRef<T>, AsMut<T>, Error, FromRow, Deduplicate, Refine.

  • Refine generates try_from and new methods for constrained types.

1.3 @scale

Hint for numeric scaling (reserved).

1.4 @variant_partition

Transforms an enum collection into separate arrays per variant (SoA with variant indexing).

1.5 @sift

Creates boolean index array for branch elimination on condition‑based iteration.

1.6 @hot / @cold

Optimisation hints for code placement.

1.7 @native

Platform‑specific implementations; compiler picks matching @native(target_os = ...).

1.8 @packed

Bit‑packed layout for struct fields (when not combined with @layout).

2. Conditional Compilation

2.1 @cfg(condition)

Includes item only if compile‑time condition true. Conditions: target_os, target_arch, feature, target_endian, any(…), all(…), not(…).

2.2 @cfg_attr(condition, attrs...)

Applies attributes only if condition holds.

3. Parser‑Facing Zero‑Cost Abstractions

3.1 @enum_consts

Generates const VARIANT = Enum::Variant; for each variant.

3.2 @tuple_struct

Generates constructor and indexing getter.

3.3 @newtype(T)

Generates from_inner, into_inner, and Deref impls.

6. Annotation Attributes (Negative Space & Hardware)

  • @copy – copyable type
  • @comptime – compile‑time only
  • @layout(...) – memory layout (soa, auto, packed, C, aosoa, adaptive)
  • @float_epsilon(...) – epsilon for ApproxEq
  • @overflow – overflow behavior (wrap, panic)
  • @must_use – warn on unused return
  • @gpu, @hardware, @parallel – effect annotations
  • @test, @bench – test/benchmark markers
  • @deprecated – deprecation warning
  • @cold, @reply, @feature, @no_std, @no_runtime, @panic("abort")
  • @slide(N), @window_nobounds, @window_accumulate
  • @debug_visualizer(closure)
  • @branchless – enforces branch‑free code; compile‑time error if branch cannot be eliminated
  • @texture(...) – asset attribute (see blaze‑asset)
  • @generate(...) – procedural asset generation
  • @streaming(...) – automatic runtime streaming
  • @cache_line(align = 64, hint = "prefetch") – cache alignment and prefetch hint
  • @aligned(16) – memory alignment
  • @volatile – prevent compiler reordering/elimination
  • @non_temporal – streaming stores (write‑combining)
  • @simd_lane(0..3) – assign to specific SIMD lane
  • @vectorize(enable = true, width = 8) – force vectorisation
  • @masked – generate masked SIMD operations
  • @device("gpu", kernel = "my_kernel") – offload to device
  • @sync_with("gpu") – synchronise across devices
  • @atomic(ordering = "release") – atomic operation ordering
  • @fence(scope = "device") – memory barrier
  • @shared – deduplicate constant across compilation units
  • @compressed(level = L, algorithm = "algo") – compress static data at build time

7. Attribute Decision Table & LSP

LSP uses pre‑computed table to suggest attributes based on patterns.

8. Testing

  • Built‑in expansions: generated code compiles and behaves correctly.
  • Conditional compilation: inclusion/exclusion with --target and --feature.
  • Parser‑facing ZCAs: constants, constructors, conversion functions.
  • Comptime macros: generated code verified.
  • Bundles: expansion correctness.
  • Annotation attributes: stored and accessible via reflection.
  • @branchless: branch‑free code enforced; compile error on failure.
  • @texture, @generate, @streaming: tested in Phase 5 (blaze‑asset).
  • Hardware attributes: alignment, SIMD, device offloading verified in assembly.
  • Negative space: constraint solver tests (compile‑time proof, runtime fallback).
