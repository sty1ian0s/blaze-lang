Blaze Phase 2e – Contracts, Invariants, and Testing Attributes

Previous: Phase 2d | Next: Phase 2f – Attributes

Goal

Implement function contracts (requires/ensures), struct invariants (contract_invariant), the @fuzz and @test_case macros, doctests, and the automatic removal of contract checks in release builds.

1. Function Contracts

1.1 Syntax

contract_clause = { "requires" expr } { "ensures" expr }
  • Each expression must be bool and pure.
  • ensures may refer to result.

1.2 Semantics

  • Debug mode: requires checked before body; ensures after body. Panic on failure with message.
  • Release mode: all contracts removed (zero runtime cost).
  • Contracts must be pure (effect set ∅).

1.3 Interaction with Linear Types

Contracts may read from parameters and result via references only; must not consume linear values.

2. Struct Invariants

2.1 Syntax

struct_invariant = "contract_invariant" expr ";"

Placed inside struct declaration, after fields. Expression may refer to self.field.

2.2 Semantics

  • Debug mode: checked after construction and after every &mut self method returns.
  • Release mode: removed.

3. Testing Attributes

3.1 @fuzz

  • Applied to a pure function with requires and ensures.
  • Syntax: @fuzz(iterations = N) (default 1,000,000).
  • Generates random inputs satisfying requires using std::random (deterministic with --reproducible).
  • Calls function, checks ensures; panics on failure.
  • Only compiled during blaze test.

3.2 @test_case

  • Applied to a @test function.
  • Syntax: @test_case([ (arg1, arg2, ..., expected), ... ]).
  • For each tuple, calls function; reports each case separately.

3.3 Doctests

  • Doc comments (///, //!) may contain Markdown code blocks.
  • Compiler extracts code blocks, compiles and runs them as tests.
  • Can reference enclosing module’s public API.

4. Release‑Mode Removal

  • requires, ensures, contract_invariant omitted in --release.
  • debug_assert! macro expands to nothing in release.
  • Overridable via blaze.toml profile settings keep_contracts and keep_assertions.

5. Implementation Notes

  • Compiler generates a wrapper function for contracts.
  • For struct invariants, checks inserted after &mut self methods and construction.
  • Fuzz harness uses reflection (later phase) to generate random values.
  • Doctest extraction via AST traversal of doc comments.

6. Testing

  • Functions with requires/ensures – verify debug execution, release removal.
  • Struct with contract_invariant – invalid construction panics in debug.
  • @fuzz – harness generated and catches bugs.
  • @test_case – all cases run, failures reported.
  • Doctests – run via blaze test.
  • Release mode – contract evaluations absent from binary/assembly.
  • Complexity: fuzz iterations configurable; linear in N.
