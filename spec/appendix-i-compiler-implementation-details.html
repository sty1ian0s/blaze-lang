Appendix I – Compiler Implementation Details

Previous: Appendix H

Status

Normative. Exact data structures, memory layouts, parallelism model, and runtime strategies that a conforming Blaze compiler must implement.

I.1 AST Storage – SoA with Typed Arenas

I.1.1 NodeId Encoding

NodeId = (kind << 24) | index. High 8 bits encode arena kind, low 24 bits encode index.

I.1.2 Node Structures

Each node kind is repr(C) with fixed‑size fields. Examples: 

struct FnDeclNode {
    name_idx: u32,
    return_type: NodeId,
    body: NodeId,
    generics: NodeId,
    with_effect: NodeId,
    where_clause: NodeId,
    contract: NodeId,
    span_start: u32,
    span_end: u32,
}
struct BlockNode {
    first_stmt: NodeId,
    span_start: u32,
    span_end: u32,
}

I.1.3 Global AST Storage

struct AstStorage {
    fn_decls: Vec<FnDeclNode>,
    struct_decls: Vec<StructDeclNode>,
    enum_decls: Vec<EnumDeclNode>,
    // … all arenas
    string_data: Vec<u8>,
    string_offsets: Vec<u32>,
}

I.2 Parallel Parsing – Chunking and Merging

Token stream split into chunks (target 64KB per chunk), aligned to token boundaries. Each chunk parsed independently; PartialAst produced. Merging: global arenas allocated, nodes copied, indices adjusted, pending items re‑parsed and stitched.

I.3 Symbol Table – Perfect Hashing

Global perfect hash function generated by blaze-perfect-hash tool. Symbol struct: 

struct Symbol {
    name_idx: u32,
    kind: u8,
    node_id: NodeId,
    next: u32,
}

Resolution: compute perfect hash, lookup table, follow next chain on collision.

I.4 Typed IR (TIR) – SoA Storage

I.4.1 Instruction Opcodes

Examples: ADD, SUB, MUL, LOAD, STORE, BR, BR_COND, PHI, CALL, RET, ALLOCA.

I.4.2 Register Allocation

Linear scan register allocator: live intervals computed over SoA arrays; registers assigned to physical registers or stack slots; spilling inserted when needed. O(instructions × registers).

I.5 @comptime Native Compilation

May compile @comptime functions to native code for speed using same backend as runtime compiler. Cached by source hash. Function must be pure. Fallback bytecode VM always available.

I.6 Memory‑Mapped AST Serialisation

.blzast binary file format: header (BZAS, version, root node id), arena headers (offset, length), arena data, string table. Mapped via mmap. Incremental reuse: unchanged AST loaded from memory map; changed files re‑parsed and merged.

I.7 Effect‑Driven Compiler Pass Scheduling

Passes declare effects (reads(ast), writes(ast), etc.). Driver builds dependency graph and executes parallel using work‑stealing scheduler. Implemented in blaze‑driver, written in Blaze and compiled AOT.

I.8 Testing the Implementation

  • AST SoA layout test: parse large file, verify nodes of same kind contiguous.
  • Parallel parsing test: 10,000 items on 16‑core machine; speedup >10x.
  • Perfect hash test: 100,000 unique identifiers; collisions <5%.
  • TIR SoA test: 10,000 instructions; opcodes array contiguous and prefetch‑able.
  • Incremental test: change one line in 10,000‑line file; recompilation <1 second.
  • Memory‑mapped AST test: load .blzast after deleting source; AST still readable and correct.
