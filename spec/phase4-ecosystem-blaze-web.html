Blaze Phase 4 – Ecosystem: Web

Previous: Wasm | Next: WGPU

Goal

Browser front‑end framework that renders Blaze GUI to a canvas via WebAssembly. No DOM; full accessibility bridge (ARIA shadow tree).

1. WebAssembly Runtime

blaze‑web‑loader.js – static JS loader that creates canvas, instantiates Wasm, forwards events, and updates ARIA live regions.

2. Integration with blaze‑gui

Same App actor; WebGpuRenderer handles web context. No code changes needed.

3. Input Handling

Events captured on canvas, translated to binary format, pushed to shared ring buffer, consumed by InputHandler actor.

4. Accessibility Bridge

Blaze serialises AccessibilityNode tree to memory buffer; JS loader polls and creates ARIA live regions.

5. Styling and Theming

Uses Style struct; no CSS.

6. Navigation and Routing

pushState and popState captured and forwarded; single‑page app routing.

7. Performance

GPU‑accelerated, flat SoA widget tree, no virtual DOM.

8. Error Handling

pub enum WebError { GpuNotAvailable, WasmLoadFailure(Text), AccessibilityBridgeFailure, Io(std::io::Error) }

9. Testing

  • Rendering: screenshot comparison (headless browser).
  • Input: simulated mouse/keyboard.
  • Accessibility: inspect ARIA subtree.
  • Routing: popState triggers UI update.
  • Performance: 1000 widgets under 8ms.
